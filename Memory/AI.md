# AI — AI 如何参与人脉跟踪

> 本文件定义**AI 在本系统中的角色、边界与领域概念**:它怎么参与跟踪流程、不能越的线、以及为支撑它而引入的数据概念。

## 一、指导思想:AI 是助手,不是裁判

AI 的价值是**降低 networking 的"启动能量"**(理工科、不擅社交 → 用系统把第一步变简单),而不是替人做关系判断。以下是不可越的线,全部长在 Domain"事实 vs 判断"之上:

1. **AI 的产物永远是"判断/草稿",绝不自动写入事实层。** 它能读资料与互动史,但**不替你打分、不擅自改阶段**。判断层(阶段/评级/当前理解)永远只由人来落。
2. **真正落库的是"人的动作"。** 典型链路:AI 起草 → 你审阅/编辑 → 你**手动发出** → 你记一条互动。**AI 草稿 ≠ 互动**;没发生的事不进时间线。
3. **不让 AI 编好感度分数。** 呼应 Domain 第五节"现实没有头顶好感度数字"——宁可让 AI 指出"可观察的行为信号",也不要它产出一个"对方好感 X 分"的伪事实。
4. **隐私边界:默认只发本地。** 联系人信息默认只发给**本地/私有部署**的模型;若要接外部模型(Claude/Gemini/OpenAI 等),**必须先有一层数据脱敏/隔离**。本轮只声明原则,脱敏层留作演进方向(见末节)。
5. **可追溯。** AI 建议应能说清"基于哪些事实"(对方资料 + 互动史 + 我的资料),便于人判断采不采信;AI 不是黑箱神谕。

> 一句话:**AI 帮你想"怎么开口",但"关系走到哪、对方什么态度"永远是你基于事实的判断。**

## 二、领域概念

### AI Provider(模型端点 · 全局 · admin 管)

一条"可调用的大模型端点"配置。设计取舍:

- **统一抽象成 OpenAI 兼容的 Chat Completions。** 绝大多数服务(本地 Ollama 的 `/v1`、OpenWebUI、OpenAI 原生)都讲这套协议;Claude/Gemini 也提供兼容层。故 v1 只实现这一种客户端。
- **留接缝不过度泛化**(呼应 Domain 第六节):配置带一个 `kind` 字段(当前 `openai-compatible`,预留 `anthropic`/`gemini`),**只在某家真不兼容时**才加专用适配器。
- **本地优先、免 key 友好:** `apiKey` 可空(本地 Ollama 无需鉴权);填了才带 `Authorization` 头。
- 全局配置(单运营者工具),由 admin 在"系统管理"页维护,可设一个默认端点供 AI 功能调用。

> 密钥存储:本轮为本地单运营者工具,`apiKey` **明文存库**。这是已知的安全弱点,接入外部付费 key 或多用户前应改为加密存储(见末节)。

### 我的资料(我 = 一个 Person + AI 人设)

AI 要个性化,就得知道"我是谁"。拆成事实与偏好两半,正好对应"事实 vs 判断":

- **我的事实背景 = 一个 Person。** 账号通过 `users.personId` 关联一条 `persons` 记录,复用人物 + 公司/学校 + 工作/教育经历的整套模型与编辑 UI——"我也是关系图里的一个节点"。我自己的 person 会从人物库列表与关系目标选择中**过滤掉**(不给自己建关系)。
- **我的 AI 人设/偏好(主观 · 账号私有)。** 一张轻量表承载用来给 AI"定调"的字段:一句话定位、我能为对方带来什么、我在找什么、语气偏好、其他常驻背景。这些是自我呈现与偏好,不是通用 Person 事实,故单独存。

于是 AI 的输入 = **我的 Person 背景 + 我的 AI 偏好 + 对方 Person 资料 + 我俩的关系与互动史**。

## 三、本轮(v1)做什么

极小可用版,只在**关系详情页**提供一个"AI 破冰建议"入口:

- 组装上述上下文 → 调默认 provider → 返回:**现在该破冰还是再观察**、**切入角度/身份**(如 CMU 校友 / 同行 / 共同项目)、**首条消息草稿**。
- 产物只到"弹窗里的可编辑草稿"为止,**不落库**;你确认后自行发送,再按现有流程手动记互动。

## 四、明确不做(本轮)

外部供应商专用适配器(先只 OpenAI 兼容)、数据脱敏/隔离层、密钥加密存储、流式输出、AI 产物历史留存、"任意阶段的下一步行动顾问"、从互动史自动给评级/下一步建议、多用户各自端点。

> 这些不是被否决,而是被排期。需要时先更新本文件,再落地。
