# DataGovernance — 数据治理:来源、可信与生命周期

> 本文件定义**数据这一维度的原则**:数据从哪来、如何保持可信、怎样贯穿其一生被对待。
> 它约束的是「怎么对待数据」这件贯穿全项目的事,而非某个具体功能;精确字段与类型以代码 schema 为准(`[../server/src/db/schema.ts](../server/src/db/schema.ts)`)。
> 与 `[Domain.md](Domain.md)`(数据长什么样)、`[AI.md](AI.md)`(AI 怎么解读数据)一脉相承——本文件管「数据如何进出与保鲜」,不重复它们的领域模型。

## 一、指导思想:系统的价值等于数据的可信度

本项目的资产不是"一堆联系人条数",而是**一批可信、可追溯、能支撑长期关系判断的数据**。数据治理就是守护这份可信度贯穿数据的一生。以下原则适用于**任何**数据(人物、关系、互动、公司/学校……),无论它是手输的、采集来的还是导入的:

1. **来源可追溯(provenance)。** 每条数据应能回答"谁录的、何时、从哪来"(现有 `createdBy/updatedBy` 审计即其一)。来路不明的数据不可信。
2. **事实与判断分层**(承接 `Domain` 第四节 / `AI`)。录进来的是事实;人或 AI 的解读是判断。二者不混淆、不互相冒充。
3. **人在回路(human-in-the-loop)。** 来自系统外部的数据,先经人确认再进事实层;不盲信任何自动化产物。自动化可以"备好草稿",落库的永远是"人的确认"。
4. **质量优先于数量。** 宁缺毋滥:价值密度 > 规模。系统是为精挑细选、长期经营而生,不追求灌满。
5. **边界与合规。** 只获取有正当理由取得的数据,尊重来源方的服务条款与隐私;风险高的动作要可弃、可控、单人自用。
6. **可迁移、不锁死。** 数据要能导出、备份、重建,不被某一次 schema 变更绑架(参见导出/备份工具,运行手册在 `[../DEV_NOTES.md](../DEV_NOTES.md)`)。

> 一句话:**先问"这条数据可不可信、从哪来、谁为它负责",再谈它能做什么。**

## 二、数据的生命周期(本维度的骨架)

治理沿数据的一生展开;每个阶段都要守住第一节的原则:

- **进入(Ingress)**:手动录入 / 半自动采集 / 批量导入。外部来源必经"人确认"这道闸。半自动采集见第三节。
- **流转与维护(Curation)**:编辑、补全、**去重与合并**、事实校正、判断(阶段/评级/理解)的更新。这是数据保鲜的日常。
- **留存与流出(Durability/Egress)**:导出、备份、重建——保证"改表不丢数据"。
- **退出(Retirement)**:归档、删除(及其连带清理)。

> 这些阶段是维度骨架:未来任何"数据怎么进/怎么保鲜/怎么带走"的能力,都应挂到这四段里,并回到第一节的原则受检。

## 三、半自动采集(Ingress 的一种实现)

规模化收录人物时,面对"零数据"瓶颈的现实取舍——它是 Ingress 的一个具体形态,不是本维度的全部:

- **由人眼筛选、脚本搬运;半自动而非全自动。** 由人在浏览时选定"要谁",脚本只把"本来会手动看的这一页"原文搬回来。价值密度优先(呼应原则 4),避开大规模自动抓取的合规与噪音风险(呼应原则 5)。
- **捕获层可替换,下游管道固定。** 采集分两段解耦:
  - *捕获层*——跑在第三方站点页面里(风险与脆弱性都在此),形态会变(油猴脚本→浏览器插件→多源适配),是后端的又一个客户端,只经 REST 契约投递。
  - *下游管道*——`原始内容 → 解析成草稿 → 人审阅/查重 → 入库`,稳定资产,与"怎么采到"无关。
- **暂存区,不是事实层。** 采集来的内容先落"入库前的前厅",经人确认(呼应原则 3)才成为人物库里的事实。原文作为可追溯证据保留(呼应原则 1)。
- **解析产物是判断,不是事实**(承接 `AI`)。从原文抽出的字段是待确认草稿,缺失留空、不编造。
- **凭证授权投递。** 捕获脚本用与账号绑定的长效凭证向后端投递,只授权"投递",不越权。

## 四、边界与延后

当前明确不做、留待需要时再启的:全自动爬虫与批量遍历、多源(Handshake/招聘站等)采集适配、**自动去重合并**(先给提示,合并沿用 `Domain` 排期)、外部模型的数据脱敏/隔离层、采集内容的多版本留存。

> 这些不是被否决,而是被排期。需要时先更新本文件把维度想清楚,再落地。
