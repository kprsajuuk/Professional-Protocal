## 一、设计基石:双层模型(客观 / 主观)

整个领域被刻意拆成两层,这是区别于"通讯录"的根本:

- **客观层(人物本身,可共享):** 这个人**是谁**——姓名、国籍、语言、教育与工作经历、公开主页。这些是事实,谁来看都一样,可由多个账号协作共建。
- **主观层(我与此人的关系,账号私有):** **我和他**之间发生了什么、走到了哪一步——关系阶段、信任、价值判断、我的私人理解与备注、互动历史。账号 A 和账号 B 对同一个人,这一层完全不同。

> 为什么这么拆:Philosophy 要求"数据模型应能承载**关系的演进过程**,而非静态联系人信息"。把"人"(静态事实)与"关系"(动态演进)分开,演进才有地方生长,归属也才清晰。

## 二、归属模型:混合(lean 实现)

- `Person`(及工作/教育经历)**共享**:任一登录账号可见、可协作编辑;记录 `createdBy`/`updatedBy` 审计。
- `Relationship`、`Interaction`**账号私有**:仅本人可见可改(owner 作用域)。
- 对某人的私有标注,当前先用 `Relationship.privateNotes` 承载。
- **推迟**:字段级私有覆盖(同一字段每账号看到不同值)、人物去重/合并。等真出现多人协作冲突再引入。

## 三、领域概念

### Person(人物 · 客观)
一个真实的 networking 目标。承载可被客观描述、可共享的资料。来源以手工/半结构化录入为主(LinkedIn/Handshake **无法合法自动抓取**)。

- **WorkExperience(工作经历):** 可多条。一段经历是复合信息:公司、职位、地点、起止、是否在职。支持足够细的职业轨迹跟踪。
- **EducationExperience(教育经历):** 可多条。学校、院系、项目/学位、专业、起止。

### Relationship(关系 · 主观 · 账号私有)
`(账号, 人)` 唯一的一条边 = "我和这个人的关系"。是核心工作面。

- **关系阶段(漏斗):** 关系演进的主轴,也是统计/跟踪的骨架。
  - `identified 识别` → `connected 已建联` → `engaged 互动中` → `trusted 信任建立` → `advocate 可引荐`
- **信任/亲密度、价值评级:** 量化"这段关系多近"与"对我的目标多重要",用于在规模化下分配有限精力。
- **可引荐潜力:** 对方是否/能向哪里引荐(对齐"高质量内推"的最终目的)。
- **当前理解(活摘要):** 随沟通不断更新的、对他项目/进展/应用场景/困难的**蒸馏认知**。区别于互动流水——它回答"现在我对他了解到哪"。
- **私人备注、相识背景、标签、状态(活跃/搁置/归档)。**

### Interaction(互动 · 账号私有 · 时间线)
关系下的一条互动记录,是"维护"动作的原子单位,也是关系演进的证据链。

- 发生时间、渠道(LinkedIn/邮件/消息/电话/会面/活动/引荐/其他)、方向(我发起/对方/双向)、内容摘要、**这次了解到了什么**、下一步。
- 设计上**只增不改语义**:互动是流水账(发生过什么);要更新"现在的认知"应改 Relationship 的当前理解。

## 四、本模型如何服务核心目的

| Philosophy 约束 | 模型对应 |
| --- | --- |
| 承载关系演进而非静态信息 | 关系阶段 + 互动时间线 + 当前理解 |
| 持续、有节奏地维护 | 互动记录(节奏/提醒为下一步增强) |
| 服务长期关系 + 双向了解 + 信任 | 信任度、当前理解、可引荐潜力 |
| 规模化不牺牲深度 | 价值/信任评级 + 阶段漏斗(便于筛选与聚焦) |

## 五、本轮(lean v1)边界:明确不做

维护节奏/下次联系提醒、价值×关系强度矩阵、目标公司联动与 H1B sponsor 情报、统计看板、LinkedIn/Handshake 自动导入、字段级私有覆盖、人物去重/合并。

> 这些不是被否决,而是被排期。需要时先更新本文件,再落地。
