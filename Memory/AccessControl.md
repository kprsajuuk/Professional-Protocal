# AccessControl — 用户与权限模型

> 本文件说明「用户 / 角色 / 访问控制」这一概念层:为什么这样设计、有哪些规则。
> 精确字段与类型见代码(`server/src/db/schema.ts`),不在此手抄。
> 最后更新:2026-06-03 ｜ 状态:活文档

---

## 它是什么 / 为什么存在

这是整个工具的「类平台地基」:在做任何主旨业务(人脉收录、关系跟踪等)之前,先有一套基本的登录与用户管理,让系统具备"谁能用、能用什么"的边界。

刻意保持简单——这不是一个完整的 RBAC / 部门 / 细粒度权限系统,把那套搬进来会偏离主线。当前只解决"有没有人能管账号、登录会不会过期、未登录会不会拦住"这些地基问题。

## 角色模型

只有两类内置角色,不做可配置的权限矩阵:

- **管理员(admin)**:可管理所有用户(增、删、改、查、重置密码)。
- **普通用户(user)**:无用户管理入口;登录后进入仪表盘等业务区域。

任意登录用户(含普通用户)都可以**自助修改自己的密码**。

## 鉴权与会话

- 登录后签发 JWT(带过期),前端存于 localStorage。
- 后端受保护接口校验 token;过期 / 无效一律返回 401。
- 前端拦截器遇 401 自动清除 token 并跳转登录页(携带回跳地址)。
- 管理员专属接口在 401 之上再加 403(非管理员访问)。

## 安全护栏(业务规则)

为避免把自己锁死或让系统失去管理员:

- 不能删除 / 停用**自己的**账号。
- 不能删除、降级或停用**最后一个管理员**(系统须至少保留一个启用的管理员)。

## 初始管理员

首次启动且用户表为空时,自动播种一个内置管理员(默认 `admin` / `admin`,可由环境变量 `SEED_ADMIN_USERNAME` / `SEED_ADMIN_PASSWORD` 覆盖)。生产环境应在首次登录后立即改密。

## 边界(当前不做)

- 不做部门 / 组织架构、细粒度资源权限、权限继承。
- 不做刷新令牌 / 多端会话管理;令牌过期即重新登录。
- 不做注册流程(用户由管理员创建)。

这些都可在未来确有需要时再迭代,届时先更新本文件再落地。
