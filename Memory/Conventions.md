## 一、文档与记录策略(hybrid)

核心问题:设计信息(比如表、字段、接口)该写成 md,还是写进代码?

结论是**分层各司其职,绝不重复**。最忌讳同一份信息存在两处(md 一份、代码一份),因为它们必然会漂移——代码改了而 md 忘了同步,过期的 md 比没有更糟。

一句话原则:

> **md 记「为什么 / 是什么意思」,代码记「具体是什么」,接口契约靠生成。**

具体分工:

| 信息类型 | 归属 | 说明 |
| --- | --- | --- |
| 概念 / 意图 | `Memory/`(md) | 领域概念定义、业务规则、设计取舍与理由。这是会演进、且代码表达不了的东西。 |
| 精确字段 / 类型 | **代码(唯一事实来源)** | 数据库 schema、迁移文件、类型定义。字段名、类型、约束以代码为准,**不再**手抄进 md。 |
| 前后端接口契约 | **代码自动生成** | 用 OpenAPI / Swagger 自动生成,或前后端共享类型定义,使契约与代码同源,永不漂移。**不手写维护**接口 md。 |

实践要点:

- **不要**为每张表、每个字段手写并维护一份 md 规范。
- **要**在 `Memory/` 里说清楚一个概念(例如「联系人」「关系阶段」「信任度」)*为什么存在、代表什么、有哪些业务规则*。
- 当需要查"某字段确切是什么类型"时,去看代码 / schema,而不是 md。

## 二、活文档维护约定

适用于 `Memory/` 下所有文档(以及 `AGENTS.md`)。

- **活文档心态。** 项目目的、概念定义、约定都会随认知深入而迭代、修正,这是预期内的,不是缺陷。
- **单一事实来源。** 某个定义或约定变化时,直接更新对应文档,不在别处另起一份,避免互相矛盾的多个版本。
- **改动留痕。** 修改任一文档时,更新其顶部的「最后更新」日期;涉及重要定义/约定变更时,在文中简要说明改了什么、为什么改。
- **先沉淀,再落地。** 新增需求或重大决策时,先写进 `Memory/`,再据此设计与编码。

## 三、层级一致性:不在聊细节时,顺手聊塌脚下的地基

> 这是一条**对话纪律**,主要约束主理人(发起讨论的人),而 agent 的职责是**帮忙发现违反并及时指出**。它不限制「能说什么」,只确保每次讨论的**地基一致**。

**核心观念:每一次讨论都有它所处的「层级 / 维度」。** 当我们在讨论某个细节(比如某个 UI、某张表、某个字段)时,指导这件事的那些文档(其归属由 [`AGENTS.md`](../AGENTS.md) 定义)就是这次讨论的**地基与原则**。地基**不是不能改**,而是**不应在聊细节的过程中被顺手改掉**。

**要改地基,先显式退到它所在的层级再谈。**

- ❌ 越级:在讨论某个 UI 体验时,聊完顺口说「把这个沉淀到 `Conventions` 吧」——但按 `AGENTS` 的定义,UI 外观/体验的沉淀属于 [`DesignGuide.md`](DesignGuide.md)。此时**不应照做**。
- ✅ 在级:若确实想改 `Conventions` 的定义本身,就站到 `AGENTS` 那一层显式地说「我认为 `Conventions` 应当包含 xxx」——此时讨论的语境已经是「在修改 Conventions」,不算越级。

**agent 的执行义务(每轮对话都要做):**

1. 收到「沉淀/修改某文档」之类请求时,先用 `AGENTS.md` 的职责定义核对:**这件事是否属于该文档的层级?**
2. 若发现**层级冲突**(请求会在当前语境下改动一块本不属于此处讨论的地基),**先指出冲突、给出正确归属**,询问是要 (a) 路由到正确的文档/层级,还是 (b) 显式退到上一层、把「修改这块地基」本身作为话题来谈。
3. 不要在语境不匹配时,默默替用户完成越级改动。

> 一句话:**只要我们脚下的石头是同一块,怎么设计、怎么改都随便聊;但若聊着把脚下的石头聊塌了,就得先退到更后面那块石头上,才能接着聊。**

## 四、工程约定

- **接口契约:标准 REST。** HTTP 状态码表达成败,成功直接返回数据,失败返回统一错误体 `{ message, ... }`(列表端点自然携带 `{ items, total }` 等 meta)。不使用 `{ code, message, data }` 信封。前端 axios 拦截器与后端 `errorHandler` 据此对齐。
- **React 只用 hooks。** 一律函数组件 + hooks,不写 class 组件。
- **不过度封装。** 现有组件已提供的能力(antd 主题、react-query 缓存等)不重复造轮子;二次封装只在确有收益(统一出口、消除重复、隔离细节)时进行。
- **分层落位。** 前端通用基础设施放 `src/` 下的 `api/lib/utils/hooks/...`,业务按域归入 `features/`;后端横切能力放 `plugins/`,业务按域归入 `modules/`。
